Claude Code の中身は固定ではありません。**同じ操作のまま、動かすモデルだけを差し替えられる**のです。難しい設計はじっくり考える賢いモデルに、ログ調べのような単純作業は速くて安いモデルに——こう振り分けるだけで、品質とコストのバランスが大きく変わります。この章では、その選び方の基礎を押さえます。

<Callout>
  モデルは <Term id="models">Opus / Sonnet / Haiku</Term> の 3 種類。賢さ・速度・コストのどれを優先するかで選びます。まずは「3 つある」「途中で替えられる」の 2 点だけ持ち帰れば十分です。
</Callout>

## §1 3 つのモデルの性格

Claude には性格の違う 3 つのモデルがあり、それぞれ得意分野が違います。難所には賢いモデル、単純作業には速いモデル、というのが基本の発想です。

| モデル | 性格 | 向くタスク |
|---|---|---|
| **Opus** | 最も賢い・難所に強い | 設計、難しいバグ、大きめのリファクタ |
| **Sonnet** | バランス型・標準 | 日常の実装、レビュー、ほとんどの作業 |
| **Haiku** | 高速・低コスト | 探索、ログ調べ、単純な繰り返し作業 |

迷ったら **Sonnet** で始めるのが無難です。バランス型なので、日々の実装やレビューはこれでほぼ困りません。

## §2 困ったら賢く、退屈なら速く

3 つを覚えても、毎回どれか迷うものです。判断はこの 2 行に集約できます。

- **手強いと感じたら Opus へ上げる** — 込み入った設計や、原因の見えないバグ。賢さがそのまま結果に効く場面です。
- **作業が単調なら Haiku へ下げる** — 大量ファイルの探索やログの拾い読み。速くて安いほうが回転します。

<Callout>
  上げ下げの基準は「**難しさ**」です。賢いモデルほど 1 回あたりのコストは高くつくので、ずっと最上位に張り付ける必要はありません。難所だけ上げて、普段は戻す——これが財布にやさしい使い方です。
</Callout>

## §3 切り替える（ターミナル / デスクトップ）

モデルは会話の途中でも切り替えられます。難所に差しかかったら上げ、抜けたら戻す、という使い方ができます。

<ModeBlock mode="terminal">
ターミナル版では <Term id="slash-command">スラッシュコマンド</Term> の `/model` を打つと、モデルの一覧から選べます。

<CodeBlock title="zsh — /model" badge="CLI">
<span class="line"><span class="sx-com"># 会話の途中でモデルを切り替える</span></span>
<span class="line"><span class="prompt cy">&gt;</span> <span class="sx-key">/model</span></span>
<span class="line"> </span>
<span class="line"><span class="sx-dim">  使うモデルを選んでください…</span></span>
<span class="line"><span class="sx-dim">    1. </span><span class="sx-fn">Opus</span>   <span class="sx-dim">最も賢い・難所向き</span></span>
<span class="line"><span class="sx-dim">    2. </span><span class="sx-fn">Sonnet</span> <span class="sx-dim">バランス型（標準）</span></span>
<span class="line"><span class="sx-dim">    3. </span><span class="sx-fn">Haiku</span>  <span class="sx-dim">高速・低コスト</span></span>
<span class="line"><span class="prompt cy">&gt;</span> <span class="caret" aria-hidden="true"></span></span>
</CodeBlock>
</ModeBlock>

<ModeBlock mode="desktop">
デスクトップアプリ版では、入力欄の近くにあるモデル選択 UI から選びます。一覧から名前をクリックするだけで、以降の応答がそのモデルに切り替わります。

ターミナルの `/model` と機能は同じです。入り口がコマンドかメニューかの違いだけで、選べるモデルも切り替わり方も変わりません。
</ModeBlock>

## §4 サブエージェントごとに決められる

選んだモデルはメインの会話に効きます。これとは別に、後の章で学ぶ**サブエージェント**には、ひとつひとつ別のモデルを割り当てられます。たとえば探索担当には Haiku、設計担当には Opus、という具合です。

その設定値のひとつが <Term id="model-inherit">model: inherit</Term> です。これは「**呼び出し元と同じモデルを使う**」という指定で、個別に固定したいときだけモデル名を書きます。いまは「サブエージェントごとにモデルを選べる」「inherit という選択肢がある」とだけ覚えておけば十分です。

<Callout>
  この「**探索は安いモデル、難所は賢いモデル**」という振り分けの考え方を <Term id="model-routing">モデルルーティング</Term> と呼びます。本章はその入り口です。サブエージェントを学ぶ第 3 部で、この考え方が本領を発揮します。
</Callout>

## §5 この章のまとめ

- モデルは **Opus（賢い）/ Sonnet（標準）/ Haiku（速い・安い）** の 3 種類。迷ったら Sonnet。
- 判断軸は「難しさ」。**難所は上げ、単調なら下げる**。普段は最上位に張り付けない。
- 切り替えは、ターミナルなら `/model`、デスクトップならモデル選択 UI。会話の途中でも替えられます。
- サブエージェントには個別のモデルを割り当て可能。`model: inherit` は呼び出し元と同じモデルを使う指定です。

次の第 3 部では、ここで触れた**サブエージェント**を本格的に作っていきます。モデルの使い分けが、そこで効いてきます。
