ここから第4部、**上級**に入ります。第3部では <Term id="subagent">サブエージェント</Term> を1体ずつ作り、呼び出せるようになりました。次の段階は、**複数のフルセッションを同時に走らせて束ねる**こと。1人の優秀な部下から、チームの編成へ——その全体像を掴むのがこの章のゴールです。

<Callout>
  <b>サブエージェントとの違いを先に。</b> サブエージェントは1つの会話の中で「別の頭」に下働きを頼む仕組みでした。ここで扱うのは、それぞれが独立した <Term id="session">セッション</Term> として並行して動く、より大きな単位の話です。
</Callout>

## §1 1体から「チーム」へ

サブエージェントを覚えると、次に欲しくなるのは「もっと並行で進めたい」という欲です。フロントエンドの修正とテストの追加とドキュメント更新を、**同時に**走らせたい。1本の会話を順番に回すのではなく、複数の作業を並べたい。

ここで登場するのが、複数のセッションを協調させる仕組みです。鍵になる考え方は3つだけ。**チームを率いるリード役**、**裏で走らせる並列実行**、**全部をまとめて見張るビュー**。この章はこの3つを順に押さえます。

## §2 エージェントチームとは

<Term id="agent-team">エージェントチーム</Term> とは、**1体の親エージェントが複数のフルセッションを束ねて動かす**運用です。親が司令塔となり、大きな仕事を小さなサブタスクに割り、それぞれを別のセッションに担当させます。

サブエージェントが「会話の中の助っ人」だったのに対し、エージェントチームは「複数の会話そのものを並べて指揮する」点が違います。規模が一段上がる、と捉えてください。

<Callout>
  <b>会社の組織図と同じ構造です。</b> リードが案件を分解して各メンバーに振り、上がってきた成果を統合する——人間のチーム運営をそのままエージェントに持ち込んだものだと考えると、腑に落ちます。
</Callout>

## §3 リードエージェントの役割

チームの中心が <Term id="lead-agent">リードエージェント</Term> です。担うのは**調整役**——自分で全部を実装するのではなく、次の3つを回します。

| 段階 | リードがすること |
|---|---|
| 分解 | 大きな依頼をサブタスクに割る |
| 割当 | 各サブタスクを別セッションに任せる |
| 統合 | 返ってきた結果をまとめ上げる |

ポイントは、リードが**手を動かす人ではなく、まとめる人**だということ。実装の細部は各メンバーに任せ、リードは全体の整合性に集中します。これは第3部で見た <Term id="plan-generate-evaluate">計画・生成・評価</Term> の関心分離を、セッションの単位に広げた形でもあります。

## §4 バックグラウンドで走らせる

複数を同時に進めるには、**ある作業を裏に回して、手元では別の作業を続ける**必要があります。これを担うのが <Term id="background-agent">バックグラウンドエージェント</Term> です。

長くかかる処理を裏へ送り、完了したら通知を受け取る。その間、自分は次の仕事に取りかかれます。リードが複数のメンバーに同時に指示を出せるのは、この「裏で走らせる」土台があるからです。

<Callout>
  <b>料理の段取りに似ています。</b> 煮込みを火にかけたまま（バックグラウンド）、横で野菜を刻む。1つが終わるのを待たずに次へ進めるから、全体が速く仕上がります。
</Callout>

## §5 ターミナルで裏へ送る

裏に回す操作は、モードで見え方が変わります。

<ModeBlock mode="terminal">
ターミナル版では、走っている処理を **`Ctrl+B`** でバックグラウンドへ送れます。送ったあとはプロンプトが戻り、別の指示を打てます。完了すると通知が出るので、長いビルドやテストを待つあいだに次の作業を進められます。

<CodeBlock title="zsh — 処理をバックグラウンドへ" badge="CLI">
<span class="line"><span class="sx-com"># 長い処理の実行中に Ctrl+B を押すと…</span></span>
<span class="line"><span class="sx-dim">  ⎿ </span><span class="sx-fn">run</span> <span class="sx-flag">npm run build</span> <span class="sx-dim">（実行中）</span></span>
<span class="line"> </span>
<span class="line"><span class="sx-dim">  ⎿ バックグラウンドへ移動しました。完了時に通知します</span></span>
<span class="line"><span class="prompt cy">&gt;</span> <span class="caret" aria-hidden="true"></span></span>
</CodeBlock>
</ModeBlock>

<ModeBlock mode="desktop">
デスクトップアプリ版では、裏に回すのは特別な操作ではありません。**新しいセッションを開けば、それぞれが独立して並行で走り**ます。1つの結果を待つあいだ、別のセッションのタブで次の作業を進められる——複数セッション並列が、もともと画面の前提になっています。
</ModeBlock>

## §6 agent view で全体を見張る

複数を同時に走らせると、今度は「どれがどこまで進んだか」が分からなくなります。それを解決するのが <Term id="agent-view">agent view</Term> です。**並行する複数のセッションを1画面でまとめて監視する**ビューで、各セッションの進捗を一覧しながらチームを運用できます。

リードが仕事を割り、メンバーが裏で走り、その全部を agent view で見渡す。この3点がそろって、はじめてエージェントチームが「目で追える」運用になります。

## §7 モードで変わる主戦場

同じエージェントチームでも、得意なモードがあります。

<ModeBlock mode="terminal">
### ターミナル版の流儀

ターミナル版は、**`Ctrl+B` で裏へ送り、ログを読みながら進める**スタイルが中心です。出力はテキストで流れるので、何が起きているかをログで追い、必要なら次の指示を差し込みます。スクリプトや自動化と組み合わせやすいのが強みです。
</ModeBlock>

<ModeBlock mode="desktop">
### デスクトップアプリ版の流儀

並列運用の**主戦場はこちら**です。<Term id="desktop">デスクトップアプリ</Term> は複数セッション並列が前提で、<Term id="agent-view">agent view</Term> で各セッションの進捗を視覚的に監視できます。どれが完了し、どれが止まっているかが一目で分かるため、チームの規模が大きいほど効いてきます。
</ModeBlock>

## §8 この章のまとめ

エージェントチームの全体像を、もう一度だけ整理します。

- **1体から複数へ** — サブエージェント（会話内の助っ人）を越え、独立した <Term id="session">セッション</Term> を並行で束ねる段階。
- **リードが調整** — <Term id="lead-agent">リードエージェント</Term> が「分解 → 割当 → 統合」を回す。手を動かす人ではなく、まとめる人。
- **裏で走らせる** — <Term id="background-agent">バックグラウンドエージェント</Term> で並列実行。ターミナルは `Ctrl+B`、デスクトップは新規セッションで。
- **まとめて見張る** — <Term id="agent-view">agent view</Term> で複数セッションを1画面監視。並列の主戦場はデスクトップ版。

全体像が掴めたら、次章ではこれをさらに深め、**サブエージェントの入れ子**と、Git の <Term id="worktree">worktree</Term> でブランチを物理的に分けて並列を安全に走らせる設計へ進みます。
