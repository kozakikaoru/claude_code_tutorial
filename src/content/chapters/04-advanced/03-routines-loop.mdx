ここまでは、あなたが指示を出してから Claude が動きました。この章で扱うのは逆——**あなたが見ていない時間に、Claude が自分で走り出す**仕組みです。朝のうちに昨日の PR をレビューしておく、夜間に CI の失敗を解析しておく。そういう「無人運転」を組みます。

<Callout>
  自動実行には大きく三つの選択肢があります。クラウドで動く <Term id="routines">ルーティン</Term>、手元のマシンで動く <Term id="scheduled-task">定期タスク</Term>、そしてセッション内で繰り返す <Term id="loop-command">/loop</Term>。動く場所と止まらない条件が違うので、まずこの違いを掴むのが本章の肝です。
</Callout>

## §1 まず「どこで動くか」で分ける

三つの自動実行は、性格がはっきり違います。選ぶ基準は **動く場所**——あなたの PC か、Anthropic のクラウドか。

| 仕組み | 動く場所 | PC を閉じても | 主な用途 |
|---|---|---|---|
| <Term id="routines">ルーティン</Term> | クラウド | **動く** | 朝の PR レビュー / 夜間 CI 解析 |
| <Term id="scheduled-task">定期タスク</Term> | 自分の PC | 止まる | ローカルのファイル/ツール操作 |
| <Term id="loop-command">/loop</Term> | 起動中のセッション内 | 止まる | 状態の定点監視（ポーリング） |

この表の一行目と二行目の差——「クラウドか、手元か」——が、すべての判断の起点になります。次の §で順に見ていきます。

## §2 ルーティン — クラウドで止まらず動く

<Term id="routines">ルーティン</Term> は、Anthropic 管理のインフラ上で動く定期実行です。最大の特徴は、**あなたの PC を閉じていても動き続ける**こと。手元のマシンの電源やネット接続に縛られません。

時刻指定だけでなく、**API や GitHub のイベントでもトリガー**できます。「毎朝 9 時」のような時間起動に加えて、「PR が立ったら」のような出来事起動も組めるということです。

<Callout>
  向くのは、止まると困る定常運用です。たとえば朝の PR レビュー、夜間の CI 失敗解析、週次の依存関係監査、マージ後のドキュメント同期。どれも「自分が起きているか」と無関係に回したい仕事ばかりです。
</Callout>

## §3 ルーティンを作る

ルーティンの作り方はモードで入り口が変わります。

<ModeBlock mode="terminal">
ターミナルでは、**`/schedule`** という <Term id="slash-command">スラッシュコマンド</Term> で作成します。対話に沿って、何を・いつ走らせるかを決めていきます。

<CodeBlock title="zsh — ルーティンを作る" badge="CLI">
<span class="line"><span class="prompt cy">&gt;</span> <span class="sx-key">/schedule</span></span>
<span class="line"> </span>
<span class="line"><span class="sx-dim">  実行する内容は? </span><span class="sx-str">"main の新しい PR をレビューして要点を報告"</span></span>
<span class="line"><span class="sx-dim">  いつ動かす?   </span><span class="sx-str">毎日 09:00</span></span>
<span class="line"><span class="sx-dim">  ⎿ ルーティンを登録しました（クラウドで実行）</span></span>
</CodeBlock>
</ModeBlock>

<ModeBlock mode="desktop">
デスクトップアプリでは、**定期タスク/ルーティンを GUI で作成・管理**できます。実行内容とスケジュールを画面のフォームに入力するだけで、クラウドセッションの起動まで設定できます。

登録したルーティンは一覧で管理でき、次回の実行予定や過去の結果もそこから確認できます。
</ModeBlock>

<Callout>
  <b>`/schedule`</b> は glossary 未収録の用語です（本文では太字で表記）。最新版でコマンド名が変わる可能性があるため、<b>使う前に公式の最新ドキュメントで確認</b>してください。
</Callout>

## §4 定期タスク — 手元で動かす

<Term id="scheduled-task">定期タスク</Term> は、決まった時刻やイベントで走らせる点はルーティンと同じですが、**動く場所があなたのマシン**です。ここが決定的な違いになります。

手元で動くからこそ、**ローカルのファイルやツールに直接アクセス**できます。クラウドからは触れない社内の開発環境、ローカルのデータベース、手元にしか無いスクリプト——そういったものを使う自動化は、定期タスクの出番です。

<Callout type="warn">
  代償は「PC を閉じると止まる」こと。電源が落ちていれば動きません。<b>止まっては困る運用はルーティン、ローカル資源が要る運用は定期タスク</b>——この線引きで選んでください。
</Callout>

## §5 /loop — セッション内で繰り返す

<Term id="loop-command">/loop</Term> は毛色が違います。これは独立した自動実行ではなく、**いま起動しているセッションの中で、同じプロンプトを一定間隔で繰り返す** <Term id="slash-command">スラッシュコマンド</Term> です。

向くのは**状態の定点監視（ポーリング）**。たとえば「デプロイが終わったか 5 分おきに確認する」「ビルドの完了を待ち続ける」といった、結果が変わるまで様子を見たい場面です。

<CodeBlock title="zsh — /loop で定点監視" badge="CLI">
<span class="line"><span class="prompt cy">&gt;</span> <span class="sx-key">/loop</span> <span class="sx-flag">5m</span> <span class="sx-str">"デプロイ状況を確認し、完了したら知らせて"</span></span>
<span class="line"> </span>
<span class="line"><span class="sx-dim">  5 分ごとに繰り返します…（Esc で停止）</span></span>
</CodeBlock>

<Callout>
  <b>継続は最大 3 日</b>とされますが、これは<b>最新版で要確認</b>です。/loop はセッションを開いている間だけ動く点も、ルーティンとの大きな違いです。
</Callout>

## §6 三つの使い分け — 早見

最後に、選び方を一枚にまとめます。考える順番は「どこで動かしたいか → 止まってよいか」です。

- **クラウドで、止まらず、定刻/イベント起動** → <Term id="routines">ルーティン</Term>（朝の PR レビュー、夜間 CI 解析）
- **手元の資源を使う定刻/イベント起動** → <Term id="scheduled-task">定期タスク</Term>（ローカル DB やローカルファイル操作）
- **起動中のセッションで結果待ちの監視** → <Term id="loop-command">/loop</Term>（デプロイ完了のポーリング）

<Callout type="warn">
  <b>自動化はコストが見えにくくなります。</b> 無人で走るぶん、気づかぬうちに <Term id="token">トークン</Term> を消費しがちです。とくに自動化経由の利用は課金枠が分かれる場合があるため、回す前に範囲とコストを見積もる癖をつけてください（課金の詳細は第5部で扱います）。
</Callout>

ここまでで「時間とイベントを起点に Claude を走らせる」運用が組めるようになりました。次章では、ターミナル・デスクトップ・スマホの間でセッションを行き来させる**サーフェス間移動とハイブリッド運用**に進み、「朝はスマホ、日中はデスクトップ、夜はターミナル」という働き方をつなげます。
