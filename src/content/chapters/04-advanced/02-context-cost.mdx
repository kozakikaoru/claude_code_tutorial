第4部もここまで来ると、複数エージェントを並列で回し、自動化まで視野に入ってきます。すると次に効いてくるのが **お金と容量の感覚** です。Claude Code は <Term id="token">トークン</Term> の量に応じてコストがかかります。賢く使う人ほど、**何を入れ、何を削るか** を意識しています。この章では、コンテキストを軽く保ち、コストを抑える道具を一通り揃えます。

<Callout>
  この章のゴールは「ケチる」ことではありません。**同じ成果を、より少ないトークンで出す**——品質を落とさずに無駄だけ削る運用へ引き上げるのが狙いです。
</Callout>

## §1 なぜコストを意識するのか

3-1 で学んだとおり、Claude は <Term id="context">コンテキスト</Term> に入っている情報を毎回見渡してから動きます。このやり取りの量が <Term id="token">トークン</Term> で測られ、**入力・出力の量がそのままコスト**になります。

会話が長引けば容れ物は埋まり、トークンは増える一方です。第4部で扱う並列実行や自動化は、放っておくとこのトークンを何倍にも膨らませます。だからこそ上級では、**コンテキストを軽く保つ技術**が品質と財布の両方を守ります。

## §2 道具は4つ

この章で扱う道具を、先に地図として並べます。順に見ていきます。

| 道具 | 効くところ |
|---|---|
| <Term id="compact">/compact</Term> ・ /clear | 膨らんだ会話を畳む |
| <Term id="plan-mode">プランモード</Term> | 実行前にコストの関所を置く |
| <Term id="model-routing">モデルルーティング</Term> | 安いモデルに振り分ける |
| <Term id="prompt-cache">プロンプトキャッシュ</Term> | 同じ前提を再利用する |

どれか1つではなく、**組み合わせて**効かせるのがコツです。まずは会話そのものを軽くする `/compact` から。

## §3 /compact ——畳んで続ける

会話が長くなってコンテキストが膨らんできたら、<Term id="compact">/compact</Term> の出番です。これは <Term id="slash-command">スラッシュコマンド</Term> の一つで、**会話の要点を残して圧縮しつつ、そのまま続ける**機能です。

過去のやり取りを Claude が要約し直し、本筋に必要な情報だけを残します。長い試行錯誤のログが畳まれ、容れ物に空きが戻る——作業を中断せずにトークンを節約できるのが利点です。

<Callout>
  目安として、ステータスラインのコンテキスト残量が少なくなってきたら `/compact` を打つ習慣を付けると、品質低下を未然に防げます。
</Callout>

## §4 /compact と /clear の違い

似た操作に <Term id="compact">/clear</Term> があります。両者は**畳む**か**捨てる**かで、はっきり別物です。

| コマンド | 何をするか | いつ使う |
|---|---|---|
| `/compact` | 要点を残して圧縮、文脈は継続 | 同じ作業を続けたいとき |
| `/clear` | 会話履歴をリセット、文脈は消える | 別の話題に切り替えるとき |

`/compact` は「机を片付けて作業を続ける」、`/clear` は「机を空にして次の案件を始める」イメージです。今の作業を続けたいのに `/clear` してしまうと、それまでの文脈を失うので注意してください。

## §5 プランモードを「コストの関所」にする

1-4 で学んだ <Term id="plan-mode">プランモード</Term> は、コスト最適化の観点でも強力な道具です。プランモードは**ファイル編集もコマンド実行もせず、何をするかの計画だけ**を立てます。

ここで方向違いに気づければ、実装に入ってから「やり直し」になる無駄を丸ごと潰せます。手戻りは、消費したトークンがそのまま捨て金になる最大の浪費です。**実行の前に計画を一度関所で止める**——これだけで、大きめのタスクのコストは目に見えて下がります。

<Callout>
  <b>関所の発想です。</b> 大きな作業ほど、いきなり走らせず計画を挟む。プランモードは「安全装置」であると同時に「コストの節約弁」でもあります。
</Callout>

## §6 安いモデルに振り分ける

次は <Term id="model-routing">モデルルーティング</Term>。2-5 で触れたとおり、<Term id="models">モデル</Term> には賢さとコストの差があります。**すべてを最も賢いモデルでやる必要はありません**。

- **探索・単純作業・実行系** → Haiku（高速・低コスト）
- **設計判断・難所** → Opus（賢いが高コスト）
- **その間のバランス** → Sonnet

「読むだけ」「ログを眺めるだけ」のような作業を高価なモデルに任せるのは、軽トラで済む荷物に大型車を出すようなもの。**仕事の重さにモデルを合わせる**のが、品質を保ったままコストを削る王道です。

## §7 探索をサブエージェントで隔離する

モデルルーティングと相性がいいのが、第3部の <Term id="subagent">サブエージェント</Term> です。横道の探索（ログ調査・大量ファイル検索）を**安いモデルのサブエージェントに丸投げ**し、本筋には<Term id="context-pollution">汚染</Term>させず**要約だけ**を返させます。

これでメインの会話は軽く保たれ、かつ探索部分は低コストのモデルで処理される——**コンテキスト温存とコスト削減を同時に**取れます。3-2 で挙げた5つの効果のうち「コンテキスト温存」と「コスト管理」が、ここで一度に効いてくる形です。

## §8 fallbackModel で止めない

もう一つ、コストと可用性に関わる設定が <Term id="fallback-model">fallbackModel</Term> です。これは**指定したモデルが使えないときに代わりに使う控えのモデル**で、複数を連ねて指定できます。

混雑や利用上限に当たっても、自動で控えのモデルに切り替えて作業を続けられます。たとえば「まず Opus、ダメなら Sonnet、それも無理なら Haiku」とチェーンしておけば、難所では賢いモデルを狙いつつ、止まらない運用にできます（連ねられる数は最大3つとされますが、最新版で確認してください）。

## §9 プロンプトキャッシュで前提を使い回す

最後に <Term id="prompt-cache">プロンプトキャッシュ</Term>。これは**繰り返し使う文脈をキャッシュして再利用する**仕組みです。

`CLAUDE.md` のルールや、毎回読ませる長い前提資料は、本来なら毎リクエスト送り直しになります。キャッシュが効けば同じ前提を送り直さずに済み、**応答が速く、コストも下がります**。基本的には自動で効きますが、「同じ大きな前提を何度も使う」運用ほど恩恵が大きい、と覚えておいてください。

## §10 モードによる見え方の違い

ここまでの道具を、実際に使うときの見え方はモードで変わります。

<ModeBlock mode="terminal">
### ターミナル版での扱い

`/compact` や `/clear` はそのまま入力欄に打ちます。モデルの切り替えは `/model`、起動時に固定したいなら `--model` フラグも使えます。コンテキストの残量や現在のモデルは <Term id="status-line">ステータスライン</Term> に表示されるので、ここを横目で見ながら「そろそろ畳むか」を判断するのが基本リズムです。
</ModeBlock>

<ModeBlock mode="desktop">
### デスクトップアプリ版での扱い

同じ機能がGUIに対応しています。デスクトップ版の強みは、**コンテキスト使用量やコストが可視化される**こと。残量メーターやコスト表示を見ながら、いつ圧縮するか・どのモデルに切り替えるかを判断できます。数字を見て直感的に「重くなってきた」と気づけるのが利点です。
</ModeBlock>

## §11 1日の運用に落とす

道具が揃ったら、日々のリズムに組み込みます。難しく考えず、次の4つを習慣にするだけで十分効きます。

- 大きめタスクは**プランモードで関所**を通してから実行する
- 会話が重くなったら **`/compact`**、話題が変わったら **`/clear`**
- 探索や単純作業は**安いモデル**（必要ならサブエージェント）に回す
- 同じ前提を繰り返すなら**キャッシュが効く構成**（`CLAUDE.md` 等）にまとめる

## §12 この章のまとめ

コンテキスト管理とコスト最適化の要点を、もう一度だけ。

- **基準はトークン** — 入力・出力の量がそのままコストになる。
- **畳む / 捨てる** — 続けるなら <Term id="compact">/compact</Term>、切り替えるなら /clear。
- **関所** — <Term id="plan-mode">プランモード</Term> で手戻りという最大の浪費を防ぐ。
- **モデルを仕事に合わせる** — <Term id="model-routing">ルーティング</Term> ＋ 探索はサブエージェントで隔離。
- **止めない・使い回す** — <Term id="fallback-model">fallbackModel</Term> と <Term id="prompt-cache">プロンプトキャッシュ</Term>。

これで「動かせる」「使いこなす」に加えて、**賢く回す**視点が手に入りました。次章では、決まった時刻やイベントで自動的に走らせる **ルーティン・定期実行** に進みます。
