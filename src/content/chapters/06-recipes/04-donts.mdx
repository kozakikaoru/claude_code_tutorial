ここまでで Claude Code をかなり任せられるようになりました。最後に、**任せてはいけない一線** を引いておきます。この章は機能紹介ではなく、事故を避けるための歯止め集です。ターミナル版でもデスクトップ版でも、ここに書く安全の考え方は同じように効きます。

<Callout type="warn">
  この章の禁止事項は、Claude Code が「できない」という話ではありません。**やれてしまうからこそ、人が線を引く** という話です。便利さと事故は背中合わせ、を前提に読んでください。
</Callout>

## §1 大原則: 取り返しのつく範囲で任せる

安全の話は、突き詰めると一文です。**元に戻せる範囲は任せる。戻せない範囲は人が最終判断する。**

コードの編集は <Term id="rewind">巻き戻し</Term> で戻せます。コミットも作り直せます。だから初級のうちから安心して試せました。一方で、本番への反映や履歴の破壊は、戻すコストが跳ね上がります。<Term id="destructive-operation">破壊的操作</Term> ——`push --force` や `rm -rf` のような取り返しのつかない操作——は、AI の自律ループの中で勝手に走らせず、人が手前で止めるのが原則です。

以降、具体的に「任せてはいけない 4 つ」を 1 つずつ見ていきます。

## §2 やってはいけない① 本番デプロイの最終判断を丸投げ

実装・テスト・ステージング確認まではどんどん任せて構いません。ですが、**本番環境へ反映する最後のボタンは人が押す** べきです。

理由は単純で、本番事故は戻すコストが最も高いからです。Claude は「テストが通った」までは判断できますが、「いま本番に出して business 影響がないか」までは見切れません。リリースのタイミング、影響範囲、ロールバック手順の確認——これらは人の責任領域です。

<Callout type="warn">
  <b>「テストが緑だから本番に出して」まで一気に頼まない。</b> デプロイ手順の自動化までは作ってよいですが、**実行のトリガーは人が引く** 設計にしてください。
</Callout>

## §3 やってはいけない② 破壊的な Git 操作を自動で走らせる

`git push --force`、`git reset --hard`、ブランチの強制削除、`rm -rf`——これらは履歴やファイルを **戻せない形で消す** 操作です。自律ループの途中でこの種のコマンドが走ると、被害に気づいたときには手遅れになりがちです。

通常のコミットや PR 作成は安全に任せられます。線を引くのは「**他人の作業や履歴を巻き込んで消す**」操作です。次の §7 で、これらを設定で遮断する方法を扱います。

<Callout type="warn">
  <b>とくに <code class="inline">push --force</code> は要注意。</b> 共有ブランチに対して実行すると、チーム全員の履歴を壊しかねません。AI に握らせず、必要なときだけ人が手で打つコマンドに分類しておきましょう。
</Callout>

## §4 やってはいけない③ セキュリティ実装を丸投げ

認証・認可・入力検証・秘密情報の扱いといった **セキュリティの要** を、「いい感じにやっておいて」で済ませてはいけません。

Claude は一般的なベストプラクティスに沿ったコードを書けますが、あなたのサービス固有の脅威モデルや、守るべきデータの重要度までは知りません。生成された認証コードが一見動いても、抜け道が残っていることはあります。**動く** と **安全** は別物です。

セキュリティに関わる部分は、Claude に書かせたうえで **人が必ずレビューする**、もしくは専門のレビュー観点を持たせた担当に二重チェックさせる。丸投げせず、最後は人の目を通す——これを徹底してください。

## §5 やってはいけない④ 曖昧なまま大規模リファクタ

「このコードベース全体をきれいにして」のような **曖昧で広範囲な依頼** は、事故の温床です。

範囲が曖昧だと、Claude は意図しないファイルまで書き換え、差分が巨大になってレビュー不能になります。どこで何が壊れたか追えなくなり、巻き戻すにも単位が大きすぎる、という最悪の状態に陥りがちです。

大きな変更こそ、まず <Term id="plan-mode">プランモード</Term> で「何をどの範囲で変えるか」を出させ、<Term id="approval-flow">承認フロー</Term> を通してから着手します。リファクタは **小さく区切って、こまめにコミット** が鉄則です。

<Callout>
  範囲を絞る・計画を先に見る・小さく刻む。この 3 つで、大規模変更の怖さはかなり消せます。「全部いっぺんに」だけ避ければよい、と覚えてください。
</Callout>

## §6 設定の落とし穴: bypassPermissions に常駐させない

<Term id="permission-mode">権限モード</Term> の 1 つ **bypassPermissions** は、確認をいっさい挟まず全操作を自動実行する「全権委任」モードです。都度の確認が煩わしくなると、つい常用したくなります。が、これは最も事故りやすい設定です。

確認をすべて飛ばすということは、§3 で挙げた破壊的操作も **黙って走る** ということです。サンドボックスなど隔離された環境での一時的な利用に留め、**自分の本番リポジトリで常駐させない**。普段は都度確認の `default`、編集だけ自動で進めたいときに限って `acceptEdits`、が安全な使い分けです。

<Callout type="warn">
  <b>「確認が面倒だから bypassPermissions」は危険信号。</b> 面倒さは、次の §7 の `deny` 設計（危ないものだけ遮断）で減らすのが正解です。全部を素通しにするのとは、安全性がまったく違います。
</Callout>

## §7 deny 設計で機密と破壊的操作を遮断する

確認の手間を減らしつつ安全を保つ鍵が、<Term id="permissions">権限設定</Term> の **deny**（完全遮断）です。<Term id="allow-deny-ask">allow / deny / ask</Term> のうち、危険なものだけを名指しで止めます。

設定ファイルに「読ませない機密ファイル」と「実行させない破壊的コマンド」を書いておけば、Claude はそこに触れようとした時点でブロックされます。これが <Term id="safety-guardrail">安全ガードレール</Term> の基本です。

<CodeBlock title=".claude/settings.json — deny の例" badge="設定">
<span class="line"><span class="sx-key">"permissions"</span>: &#123;</span>
<span class="line">  <span class="sx-key">"deny"</span>: [</span>
<span class="line">    <span class="sx-str">"Read(./.env)"</span>,             <span class="sx-com">// 秘密情報を読ませない</span></span>
<span class="line">    <span class="sx-str">"Read(./secrets/&#42;&#42;)"</span>,</span>
<span class="line">    <span class="sx-str">"Bash(git push --force&#42;)"</span>, <span class="sx-com">// 強制 push を遮断</span></span>
<span class="line">    <span class="sx-str">"Bash(rm -rf&#42;)"</span></span>
<span class="line">  ]</span>
<span class="line">&#125;</span>
</CodeBlock>

危ないものを `deny` で塞いでおけば、残りは安心して自動化できます。「全部止める」のではなく「**危険だけ止める**」——これが現実的な落としどころです。

## §8 課金事故を防ぐ（クレジット制に注意）

自動化を勧めてきた以上、コスト事故にも触れておきます。**2026/6/15 から、自動化経由の利用は通常のサブスク枠とは別枠** になりました。

具体的には、`claude -p`（ヘッドレス実行）・Agent SDK・GitHub Actions・SDK 経由のアプリは、<Term id="monthly-credit">月次クレジット</Term> を消費します（Pro / Max などのプラン別枠、超過分は API レート課金）。対話で普通に使う分とは課金の出どころが違う、という点が見落とされがちです。

危ないのは、**定期実行や `/loop` を回したまま放置** するパターンです。気づかないうちにクレジットを使い切る、あるいは API 課金が膨らむことがあります。

<Callout type="warn">
  <b>自動化を仕込んだら、止め方と上限をセットで決める。</b> 不要になったルーティンは止める、ループは期限を切る、利用量を定期的に確認する。「動かしっぱなし」が最大のコスト事故要因です。
</Callout>

## §9 まとめ: 最後に引く 4 本の線

この章を 1 枚にまとめます。迷ったらここに戻ってください。

| 領域 | やってはいけない | 代わりにこうする |
|---|---|---|
| デプロイ | 本番反映を丸投げ | 手順は自動化、実行は人が押す |
| Git | 破壊的操作を自動実行 | `deny` で遮断、必要時だけ手動 |
| セキュリティ | 認証・検証を丸投げ | 書かせて人が必ずレビュー |
| 変更範囲 | 曖昧なまま大規模リファクタ | プランモード→小さく刻む |

設定面では **bypassPermissions に常駐しない / `deny` で危険だけ塞ぐ / 自動化の課金枠と止め方を管理する**。この歯止めさえ守れば、ここまで学んだ機能を安心して攻めに使えます。便利さを最大化する鍵は、実は **安全の線をはっきり引いておくこと** なのです。
