第5部では `claude -p`・GitHub Actions・<Term id="agent-sdk">Agent SDK</Term> と、Claude を自動で走らせる手段を次々に紹介してきました。便利になるほど、見落とせなくなるのが **コストと権限の事故** です。本章は、自動化を勧めた責任として **お金の事故** と **権限の事故** を防ぐルールを固めます。

<Callout>
  この章はモードによる差がありません。ここで話すのは運用ポリシー——ターミナル版でもデスクトップ版でも、チームで決めて守るべき共通の約束ごとです。
</Callout>

## §1 まず知るべき課金の変更

2026/6/15 から、Claude Code の課金に大きな変更が入りました。ひとことで言うと——**自動化経由の利用は、通常のサブスク枠とは別の枠を使う** ようになりました。

これまで「Pro / Max のサブスクに入っていれば対話で使い放題（上限内）」という感覚で使えていた人ほど、ここは意識を切り替える必要があります。自動化を本格的に回す前に、この一点だけは必ず押さえてください。

<Callout type="warn">
  <b>料金の具体額は必ず最新版で確認してください。</b> 本章の数値は <b>2026-06 時点のスナップショット</b>です。Claude Code の料金体系は更新が速く、プラン名・金額・上限は変わり得ます。導入前に公式の料金ページで最新を確認するのが大前提です。
</Callout>

## §2 何が「別枠」になるのか

別枠（<Term id="monthly-credit">月次クレジット</Term>）を消費するのは、対話セッションではなく **自動化の入口** です。次の4つが対象だと覚えてください。

| 利用形態 | クレジット消費 |
|---|---|
| 対話セッション（`claude` で会話） | 通常のサブスク枠 |
| <Term id="claude-p">claude -p</Term>（ヘッドレス実行） | 別枠の月次クレジット |
| <Term id="github-actions">GitHub Actions</Term> 連携 | 別枠の月次クレジット |
| <Term id="agent-sdk">Agent SDK</Term> / SDK 経由アプリ | 別枠の月次クレジット |

線引きはシンプルです。**人が画面の前で対話する** なら通常枠、**仕組みが自動で走らせる** なら別枠。第5部で学んだ自動化の手段は、ほぼすべて後者にあたります。

## §3 プランと枠の関係

別枠の月次クレジットは、加入しているプランに応じて付与されます。2026-06 時点では、おおむね次の構成です（金額は必ず最新版で確認）。

| プラン | 月額（2026-06 時点・要確認） |
|---|---|
| Pro | $20 |
| Max 5x | $100 |
| Max 20x | $200 |

この月次クレジットを使い切った後は、**API レート課金**（使った分だけの従量課金）に切り替わります。つまり「サブスクに入っているから青天井で自動化してよい」わけではない、という点が重要です。

<Callout type="warn">
  <b>金額・プラン名は変わります。</b> 上の表はあくまで 2026-06 のスナップショットです。最新の正確な数値は、運用を始める前に公式の料金ページで確かめてください。本サイトの数値で予算を確定させないこと。
</Callout>

## §4 なぜ自動化章すべてに注意書きがあるのか

第4部・第5部の自動化を扱う章には、必ずコストの注意喚起が添えてありました。その根拠がこの課金変更です。

対話なら、コストは「自分が打ち込んだ分」に比例します。ところが <Term id="claude-p">claude -p</Term> を <Term id="cron">cron</Term> で毎朝走らせたり、GitHub Actions を全 PR に効かせたりすると、**人が見ていない間に <Term id="token">トークン</Term> が積み上がります**。気づいたら別枠を使い切っていた、という事故が起きやすい。

だからこそ、自動化を仕込むときは「これは別枠を、どのくらいの頻度で消費するか?」を先に見積もる癖をつけてください。

## §5 コスト事故を防ぐ運用

具体的な歯止めを4つ挙げます。難しいことはなく、**走らせる前にひと呼吸** が基本です。

- **頻度を絞る** — 定期実行は「本当に毎日要るか?」を問う。週次で足りるなら週次に。
- **対象を絞る** — 全 PR ではなく、特定ラベルや特定パスの変更だけに効かせる。
- **モデルを選ぶ** — 自動化の定型処理は <Term id="models">Haiku</Term> など軽いモデルへ。<Term id="model-routing">モデルルーティング</Term> はコスト対策でもあります。
- **使用量を見る** — 別枠の消費量を定期的に確認し、想定とずれていないか点検する。

「便利だから全部自動化」ではなく、効く所だけに絞る——コストはこの一手で大きく変わります。

## §6 もうひとつの事故＝権限

コストと並ぶもうひとつのリスクが **権限の事故** です。自動化されたエージェントは、人の確認を挟まずにコマンドを実行できてしまう場面があります。だからこそ、与える権限を最初から絞っておく必要があります。

ここで効いてくるのが、第3部から繰り返してきた原則——<Term id="least-privilege">最小権限</Term> です。**各エージェント・各ツールに、必要最小限の権限しか与えない**。自動化の規模が大きくなるほど、この原則の重みが増します。

<Callout>
  最小権限は「不便にする」ための制約ではありません。<b>事故が起きても被害を小さく閉じ込める</b>ための設計です。狭く始めて、足りなければ足す——この順番を守ります。
</Callout>

## §7 最小権限を仕込む2つの場所

最小権限は、主に2か所で具体化します。どちらも第2〜4部で触れた仕組みの応用です。

- **サブエージェントのスコープ権限** — 各サブエージェントの <Term id="tool-permission">ツール権限</Term> を役割ぶんだけに絞る。調査役に編集権限は渡さない、評価役にコマンド実行は渡さない。
- **settings.json の <Term id="permissions">permissions</Term>** — <Term id="allow-deny-ask">allow / deny / ask</Term> で全体のガードを張る。とくに危険な操作や機密ファイルは <b>deny</b> で完全に塞ぐ。

「許可リストを広げる」より「**禁止リストを先に固める**」発想が、自動化では効きます。

## §8 deny で固める例

権限設計の勘どころは、`deny` を先に書くことです。`allow` を緩めても、`deny` に当たれば止まる——この優先順を使って、取り返しのつかない操作を塞ぎます。

<CodeBlock title="settings.json — deny を先に固める" badge="設定">
<span class="line"><span class="sx-key">"permissions"</span>: &#123;</span>
<span class="line">  <span class="sx-key">"deny"</span>: [</span>
<span class="line">    <span class="sx-str">"Bash(git push --force*)"</span>,   <span class="sx-com">// 破壊的な force push を塞ぐ</span></span>
<span class="line">    <span class="sx-str">"Bash(rm -rf*)"</span>,             <span class="sx-com">// 一括削除を塞ぐ</span></span>
<span class="line">    <span class="sx-str">"Read(./.env*)"</span>              <span class="sx-com">// 秘密情報の読み取りを塞ぐ</span></span>
<span class="line">  ]</span>
<span class="line">&#125;</span>
</CodeBlock>

書式は環境やバージョンで変わり得ます。**最新版の書き方は公式ドキュメントで確認** してください。大事なのは「危険な操作を名指しで `deny` する」という発想です。

## §9 bypassPermissions の扱い

<Term id="permission-mode">権限モード</Term> の中で、最も注意が要るのが **bypassPermissions** です。これは確認をすべて飛ばす「全権委任」モード——便利に見えて、自動化と組み合わせると最も危険な設定です。

人が画面を見ていない自動実行で全権委任にすると、暴走を止める最後の関所が無くなります。

<Callout type="warn">
  <b>自動化で bypassPermissions を常用しないこと。</b> 確認を飛ばしたいなら、まず <b>deny で危険操作を塞いだ上で</b>、必要な範囲だけ <Term id="allow-deny-ask">allow</Term> する設計に置き換えます。「全部 OK」ではなく「ここだけ OK」へ。
</Callout>

## §10 AI に任せてはいけない領域

最後に、いちばん大事な線引きです。どれだけ自動化が進んでも、**人間が最終判断を持つべき領域** があります。次の3つは、AI に丸投げしてはいけません。

- **本番デプロイの最終判断** — 出すか出さないかは人が決める。自動化は準備までに留める。
- **<Term id="destructive-operation">破壊的な Git 操作</Term>** — `push --force` や履歴の書き換えなど、取り返しのつかない操作。
- **セキュリティの丸投げ** — 認証・認可・秘密情報の扱いを「よしなに」で任せない。

<Callout type="warn">
  <b>ここは妥協してはいけません。</b> 「戻せない・人に影響が及ぶ・組織の信頼に関わる」操作は、自動化の対象外と決めておきます。AI は強力な実行者ですが、<b>責任を負うのは人間</b>です。やってはいけない集は第6部 6-5 でさらに詳しく扱います。
</Callout>

## §11 ガバナンスとしてまとめる

ここまでのコスト対策と権限対策を、チームの **<Term id="governance">ガバナンス</Term>** として明文化します。個人の心がけに頼らず、ルールとして共有するのが要点です。

- [ ] 自動化が **別枠の <Term id="monthly-credit">月次クレジット</Term>** を使うことをチーム全員が理解している
- [ ] 料金・上限を **最新版で確認** し、予算と頻度を決めてある
- [ ] サブエージェントとツールに **<Term id="least-privilege">最小権限</Term>** を適用している
- [ ] 危険操作・機密ファイルを **<Term id="permissions">deny</Term>** で塞いである
- [ ] 自動化で **bypassPermissions を常用していない**
- [ ] **AI に任せない領域**（本番デプロイ判断・破壊的 Git・セキュリティ）を決めてある

<Callout>
  <b>これがプロの到達点です。</b> 自動化の威力を引き出しながら、コストと権限を統制下に置く——ここまで設計できれば、Claude Code を「個人の道具」から「チームで安全に回せる仕組み」へと引き上げられています。
</Callout>
