第3部もここで最後です。サブエージェントの作り方、`description` の設計、3分離パターン、フックや MCP——道具は一通り揃いました。本章では、それらを**毎日の運用にどう落とすか**をまとめ、最後に「ここまでできれば中級者」という自己診断チェックリストを置きます。

<Callout>
  この章はモードによる差がありません。ここで話す運用の考え方は、ターミナル版でもデスクトップ版でも同じように効きます。
</Callout>

## §1 まず立ち返る一点

道具が増えると、つい「何でもサブエージェント化したくなる」誘惑が出てきます。けれど立ち返るべき原点は一つでした——<Term id="subagent">サブエージェント</Term>は**コンテキストを守るための仕組み**だということ。

多機能だから使うのではありません。本筋の容れ物を汚さないために、横道の作業を**別の頭に逃がす**。この一点さえ外さなければ、運用の判断はおおむねぶれません。以降の判断軸も、すべてここから導けます。

## §2 いつサブエージェント化すべきか

作るかどうか迷ったら、次の表で当たりを付けます。要は、**繰り返し使う**か、**メイン会話を汚す**か、**権限を絞りたい**か——どれかに当てはまれば作る価値があります。

| 作る価値あり | 作らなくてよい |
|---|---|
| ログ調査・大量探索など本筋を汚す作業 | 一度きりの単純な指示 |
| レビューやテストなど繰り返す定型作業 | メイン会話のままで十分短い作業 |
| 危険ツールを使わせたくない作業 | 専門化する必要のない雑多な依頼 |

迷ったときの合言葉は <Term id="single-responsibility">単一責務</Term>です。「この子の仕事は一言で言える?」と問うてみて、言えないなら、まだ分け方が粗いサインです。

## §3 description の見直し

サブエージェントが**呼ばれない**——中級者がいちばん多く踏む小石がこれです。原因のほとんどは <Term id="description">description</Term> にあります。<Term id="auto-delegation">自動委譲</Term> は説明文を手がかりに動くからです。

- 呼ばれてほしい場面で呼ばれない → トリガー語・対象・場面を**具体的に**書き足す
- 関係ないのに呼ばれる → 守備範囲を絞り、紛らわしい語を削る

「思った相手が呼ばれているか」を、ときどき実際のタスクで確かめる。これが最初の見直しポイントです。

## §4 ツール権限の見直し

次に確かめるのは、各サブエージェントの <Term id="tool-permission">ツール権限</Term> です。守るべき原則は <Term id="least-privilege">最小権限</Term>——**その役割に必要なツールだけ**を与えます。

調査役に編集権限はいりません。レビュー役にコマンド実行はいりません。権限を絞るほど事故が減り、しかも**挙動が安定する**という副産物まで付いてきます。「この子に、この権限は本当に要るか?」を定期的に棚卸ししてください。

<Callout type="warn">
  権限は「とりあえず全部許可」が最も危険です。困ったら広げるのは簡単ですが、広げたまま忘れるのが事故の元。<b>狭く始めて、足りなければ足す</b>——この順番を守ります。
</Callout>

## §5 モデルの見直し

最後はモデルの割り当てです。考え方は <Term id="model-routing">モデルルーティング</Term>——**探索や実行系は軽いモデル、難所だけ賢いモデル**に振り分けます。

| 役割の例 | 目安のモデル |
|---|---|
| 調査・テスト実行など軽作業 | Haiku（速い・安い） |
| 一般的な実装・レビュー | Sonnet（バランス） |
| 難所の計画・設計 | Opus（最も賢い） |

すべてを最上位モデルで回すと、品質は上がってもコストが膨らみます。役割に対してモデルが過剰・過小になっていないか、ときどき見直すとコストが締まります。

## §6 チームで共有する

ここまで作り込んだ資産は、自分一人で抱えるにはもったいないものです。サブエージェントの実体は <Term id="dot-claude">.claude フォルダ</Term> の中のファイルでした。つまり、**git にコミットすればチーム全員に配れます**。

- `.claude/agents/` をコミット → チームで同じサブエージェントを共有
- 個人専用にしたいものは `~/.claude/agents/`(個人用)に置いて分ける

「自分の手元でだけ賢い Claude」から「**チーム全員で賢い Claude**」へ。ここから先の本格的なチーム運用は、第5部であらためて扱います。

## §7 中級到達チェックリスト

仕上げに自己診断です。次がひと通り「自分の言葉で説明でき、実際にできる」なら、あなたはもう中級者です。

- [ ] **コンテキストが命**——サブエージェントがコンテキストを守る仕組みだと説明できる
- [ ] <Term id="subagent">サブエージェント</Term>を一つ作り、<Term id="description">description</Term> で狙い通りに呼べる
- [ ] <Term id="tool-permission">ツール権限</Term>を <Term id="least-privilege">最小権限</Term>で絞れる
- [ ] タスクに応じて <Term id="model-routing">モデル</Term>を使い分けられる
- [ ] <Term id="plan-generate-evaluate">計画・生成・評価</Term>の3分離パターンを説明できる
- [ ] <Term id="hook">フック</Term>で編集後の整形や危険コマンドの遮断を仕込める
- [ ] <Term id="mcp">MCP</Term> で外部ツールを Claude に操作させられる

<Callout>
  <b>到達点に立ちました。</b> ここまでで Claude Code は「実務にかなり投入できる」段階です。第4部からは、複数エージェントの並列運用や自動化へ——一流(上級・プロ)への階段を上っていきます。
</Callout>
